# สถาปัตยกรรมระบบรายงานสถานการณ์ฝุ่นละออง PM2.5 สคพ.11

ระบบรายงานสถานการณ์ฝุ่นละออง PM2.5 ของ สคพ.11 ถูกออกแบบและพัฒนาขึ้นภายใต้แนวคิด **Serverless Architecture** โดยเป็นเว็บแอปพลิเคชันรูปแบบ **Single Page Application (SPA.)** ที่ไม่มีเซิร์ฟเวอร์ประมวลผลหลังบ้านเป็นของตัวเอง อาศัยการประมวลผลบนเบราว์เซอร์ของผู้ใช้งานและการเชื่อมต่อกับบริการภายนอก (Third-party APIs.) อย่างเต็มรูปแบบ ตอบโจทย์การใช้งานระดับบริหารที่ต้องการความคล่องตัวสูง นำข้อมูลไปใช้ประกอบการตัดสินใจและสั่งการได้อย่างรวดเร็ว โดยไม่ต้องรับภาระค่าใช้จ่ายและการบำรุงรักษาระบบเซิร์ฟเวอร์

จากการพัฒนาและปรับปรุงอย่างต่อเนื่อง ระบบได้ก้าวข้ามข้อจำกัดของการประมวลผลฝั่งไคลเอนต์ (Heavy Client) มาสู่สถาปัตยกรรมแบบกระจายภาระงานที่สมบูรณ์แบบ โดยสามารถแบ่งองค์ประกอบหลักได้ดังนี้

## 1. ส่วนแสดงผลหน้าบ้าน (Frontend หรือ Client-side)
พัฒนาด้วยเทคโนโลยี HTML และ Tailwind CSS ควบคู่กับ JavaScript โหลดหน้าเว็บได้รวดเร็วและตอบสนองไวเนื่องจากทำงานแบบ SPA. และมีความเบา (Lightweight) หลังจากการย้ายตรรกะที่ซับซ้อนไปไว้หลังบ้าน
* **โครงสร้างและหน้าตาเว็บ (UI.):** ใช้ HTML5 และ Tailwind CSS ผ่านไฟล์คอมไพล์สำเร็จรูป รวมถึงการเขียน CSS แบบ Inline เพิ่มเติมเพื่อจัดการสีของค่า AQI. และแอนิเมชันให้เหมาะสมกับการนำเสนอข้อมูล
* **การแสดงผลกราฟิกและข้อมูล:** ใช้ไลบรารี Chart.js และ Chartjs-plugin-datalabels สำหรับสร้างกราฟแท่งเปรียบเทียบค่าฝุ่นและจุดความร้อนสะสม
* **การรวบรวมสถานการณ์:** หน้าบ้านมีหน้าที่รับข้อมูลที่ประมวลผลสำเร็จรูปแล้วจาก API. หลังบ้านมาแสดงผล และรวบรวมข้อมูลสถานการณ์ปัจจุบันเพื่อสร้างเป็นข้อความคำสั่ง (Prompt) ส่งให้ AI. ประมวลผล

## 2. ส่วนประมวลผลหลังบ้าน (Backend และ Middleware)
ใช้ Google Apps Script (GAS.) ทำหน้าที่เป็น Backend-for-Frontend (BFF.) หรือตัวกลางที่ช่วยจัดการตรรกะทางธุรกิจที่ซับซ้อน ลดภาระการทำงานของอุปกรณ์สมาร์ทโฟนและยกระดับความปลอดภัยขั้นสูงสุด
* **Data Proxy:** ดึงข้อมูลประวัติ PM2.5 และโควตาวันฝุ่นเกินมาตรฐานจากตารางข้อมูลใน Google Sheets
* **Geospatial Processing:** ซ่อน API Key ของ NASA ไว้หลังบ้านเพื่อความปลอดภัย ดึงไฟล์พิกัดโพลีกอนขอบเขตประเทศไทย และประมวลผลจุดความร้อนด้วยสมการคณิตศาสตร์แทนการโหลดไลบรารีบนหน้าเว็บ แล้วส่งผลลัพธ์กลับมาเฉพาะตัวเลขสรุปของพื้นที่ นครราชสีมา ชัยภูมิ บุรีรัมย์ และสุรินทร์
* **Weather Proxy:** ดึงข้อมูลสภาพอากาศและโอกาสเกิดฝนจากกรมอุตุนิยมวิทยา (TMD API.) เพื่อแก้ไขปัญหาข้อจำกัดเรื่อง CORS. ของเบราว์เซอร์
* **AI Proxy:** รับคำสั่งข้อความและดึงรูปภาพกราฟอัตราการระบายอากาศ (VR.) ของทั้ง 4 พื้นที่มาแปลงเป็นรหัส Base64 แล้วส่งต่อไปยัง Gemini API. เพื่อวิเคราะห์และร่างข้อความอัจฉริยะสำหรับส่งเข้ากลุ่ม LINE. เพื่อสั่งการต่อไป

## 3. ส่วนฐานข้อมูลและบริการภายนอก (Data Sources และ Third-party APIs.)
* **ฐานข้อมูลหลัก:** ใช้ Google Sheets ทำหน้าที่เป็นระบบจัดการฐานข้อมูล (DBMS.) ที่มีความยืดหยุ่นสูงและทีมงานสามารถอัปเดตข้อมูลได้ง่าย
* **บริการข้อมูลจุดความร้อน:** เชื่อมต่อกับ NASA FIRMS API.
* **บริการข้อมูลสภาพอากาศและภาพกราฟ:** ดึงข้อมูลเรียลไทม์จากเซิร์ฟเวอร์ของกรมอุตุนิยมวิทยา รวมถึงใช้วิธีสร้างลิงก์แบบไดนามิกเพื่อดาวน์โหลดรายงาน PDF. จากระบบ Air4Thai ของกรมควบคุมมลพิษ (คพ.)
* **ระบบผู้ช่วยอัจฉริยะ:** ประยุกต์ใช้ Gemini API. ในการเป็นผู้ช่วยวิเคราะห์ข้อมูลและร่างข้อความ

## บทสรุปการยกระดับสถาปัตยกรรม
ระบบนี้ถูกพัฒนาและต่อยอดจากการประมวลผลทุกอย่างด้วยเครื่องของผู้ใช้ มาเป็นการใช้ประโยชน์จาก Cloud Services อย่างเต็มรูปแบบ การย้ายตรรกะการคำนวณพิกัดและการเชื่อมต่อ API. ภายนอกไปไว้ที่ GAS. ช่วยให้ระบบมีความปลอดภัยของข้อมูลระดับสูง โหลดข้อมูลได้เร็วกว่าเดิม ลดปัญหาเครื่องค้างเมื่อมีจุดความร้อนปริมาณมาก และพร้อมสำหรับการขยายขีดความสามารถเพื่อรองรับภารกิจการจัดการสิ่งแวดล้อมเชิงรุกได้อย่างยั่งยืน
